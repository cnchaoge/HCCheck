### Step 2.4：抽 `extract_plates_from_table(table)`

**应用场景：**

- `main()` 里扫描工作台 table

**新代码：**
```python
def extract_plates_from_table(table, skip_plates=None):
    """从 table 中提取所有车牌号(可选过滤黑名单)
    Returns: [(plate, row_text, cells), ...]
    """
    if skip_plates is None:
        skip_plates = SKIP_PLATES
    results = []
    try:
        rows = table.locator("tbody tr")
        for r in range(rows.count()):
            cells = rows.nth(r).locator("td")
            row_texts = [cells.nth(j).text_content().strip() for j in range(cells.count())]
            row_full = " ".join(row_texts)
            for t in row_texts:
                if config.PLATE_RE.match(t) and t not in skip_plates:
                    results.append((t, row_full, cells))
                    break
    except:
        pass
    return results
```

```python
def _process_one_car(page, plate, current_node, flow_name, processed, failures):
    """处理一辆车（含重试 + 黑名单逻辑）"""
    step_map = {
        config.NODE_VEHICLE_CHECK: config.STEP_VEHICLE_CHECK,
        ...
    }
    run_from_step = step_map.get(current_node, "unknown")
    processed += 1
    print(f"\n{'='*55}")
    print(f"  第 {processed} 辆: {plate}")
    try:
        dispatch(page, plate, run_from_step, processed)
        print(f"\n  ✅ {plate} 全部流程完成")
        failures.pop(plate, None)
    except Exception as e:
        failures[plate] = failures.get(plate, 0) + 1
        fc = failures[plate]
        print(f"\n  ❌ {plate} 处理失败 ({fc}/{config.MAX_FAIL}): {e}")
        screenshot_on_error(page, plate)
        _close_all_popups(page)
        if fc >= config.MAX_FAIL:
            SKIP_PLATES.add(plate)
            print(f"  ⏭ {plate} 已加入黑名单,不再自动重试")
    pa(1.5)
    return processed
```

```python
def _process_new_plate(page, processed, failures):
    """去列表取新车并处理
    Returns: (processed, should_break)
    """
    # 导航
    contents = get_contents(page)
    try:
        _click_expand_freight_manage(contents)
        _click_normal_review_link(contents)
        if not _verify_in_normal_review(page):
            _click_normal_review_link(contents)
    except Exception as e:
        print(f"  普货审验自动导航失败: {e}")
        input("  >>> 请手动点左侧[普货审验],然后按回车...")
    
    # 取车牌
    plate = get_next_plate_from_list(page)
    if not plate:
        return processed, False  # 列表空
    
    # 黑名单检查
    if plate in SKIP_PLATES:
        print(f"  ⏭ 跳过 {plate} (已被拉黑)")
        return processed, False
    
    # 处理
    processed = _process_one_car(page, plate, "", "", processed, failures)
    
    # 单次/上限检查
    if config.SINGLE_RUN:
        print("\n  🛑 单次模式,已处理 1 辆,停止")
        return processed, True  # 停止
    if config.MAX_CARS > 0 and processed >= config.MAX_CARS:
        print(f"\n  🛑 已达上限 {config.MAX_CARS} 辆,停止")
        return processed, True  # 停止
    
    return processed, False
```

## 🚦 健壮性优先级（新方向，优先推进）

### #1 智能等待 ✅ 已完成

**背景：** `pa(2)` / `pa(3)` 是猜时间。页面加载慢时不够，加载快时浪费。

**方案：**
- `utils.wait_until(condition, timeout, poll, description, screenshot_page)` — 轮询条件满足或超时
- `utils.wait_until_not(condition, ...)` — 轮询条件消失或超时
- `config.WAIT_UNTIL_TIMEOUT=10.0` / `config.WAIT_UNTIL_POLL=0.3` 默认常量

**使用现状：**
- `commit ea3851c` 已交付 `wait_until` / `wait_until_not` 工具函数
- `commit f2d6c2f` popup4 retry 之间已替换为 `wait_until(_any_year_check_visible, timeout=3, poll=0.5)`
- 运管站电脑验证：12:30 跑 17 辆 popup4 仍偶发 retry，但弹窗残留导致后续错乱（见 #4），不是 wait_until 本身问题

### #2 异常分类 ⏳ 待做

**背景：** 所有异常都走 `except Exception`，重试/黑名单/人工介入逻辑散落各处。

**方案：**
```python
class RecoverableError(Exception): ...    # 网络/超时，重试
class PermanentError(Exception): ...      # 业务规则（不能创建流程），直接拉黑
class UserInterveneError(Exception): ...  # 验证码/页面变更，要求人工
```

业务层抛对应异常，上层调度器按类型走不同分支。**预计改动**：新建 `errors.py` + 改 `popups/*.py` + 改 `run.py` 上层分支，~80 行。**风险**：中，需多场景验证。

### #3 启动清理 ⏳ 待做

**背景：** 上次崩溃后可能遗留 chromium 进程 / 临时文件 / 状态变量，本次启动可能被污染。

**方案：**
- 启动时检查遗留 chromium 进程 → kill
- 启动时清临时文件、cookie、context 缓存
- 启动时验证 `.skip_plates.json` / `.user_config.json` 格式

**预计改动**：~30 行 helper + 启动逻辑接入。**风险**：低。

### #4 popup4 retry 弹窗残留修复 ⏳ 待做（**2026-06-24 12:30 发现**）

**背景：** popup4 触发 retry 时，**前一轮的 submitDiag 弹窗可能没关掉**，dialog handler 处理了**旧弹窗**，**新弹窗**没人处理。结果：
- 工作台节点停留在 `business_review`（未真正提交）
- 但脚本以为到了 `archive`，错进 popup3
- popup3 在残留弹窗里找元素，全找不到 → 失败 1/3
- “清理 2 个残留弹窗”证明了多开现象

**修复方向（未实施）：**
- popup4 retry 之前先 `wait_until_not(弹窗存在, timeout=3)` + 强制关残留
- popup4 提交后加状态断言（assert 工作台节点变 `archive`），不达预期则抛异常走人工介入
- 加 `_any_year_check_visible` 类似 helper：检查 popup 是否有未处理的 submitDiag 弹窗

**预计改动**：~40 行。**风险**：低（增强而非重构）。**价值**：高（直接消除当前最大隐患）。
